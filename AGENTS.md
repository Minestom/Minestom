# AGENTS.md

Guidance for AI coding agents working in this repository.

## What this is

Minestom is a Java library for building Minecraft server software from scratch — it implements the Minecraft protocol but ships **no vanilla gameplay**. Users embed it like any other library; there's no plugin/runtime distribution. Java 25 toolchain, Gradle (Kotlin DSL), multi-module build.

## Common commands

```bash
./gradlew build                                  # compile everything + run tests
./gradlew :test                                  # run only the main module's tests
./gradlew :test --tests "net.minestom.server.entity.EntityTest"   # one class
./gradlew :test --tests "*.EntityTest.basic"                      # one method
./gradlew :demo:runShadow                        # launch the demo server (localhost:25565)
./gradlew :jmh-benchmarks:jmh                    # run JMH benchmarks
./gradlew :jcstress-tests:jcstress               # concurrency stress tests
./gradlew determineMinecraftVersion              # print the targeted MC version
```

Tests run with these JVM flags set by `minestom.java-library.gradle.kts`: `-Dminestom.viewable-packet=false`, `-Dminestom.inside-test=true`, `-Dminestom.acquirable-strict=true`. Do not assume "normal" production behavior in tests for those toggles.

## Modules

- **root (`net.minestom:minestom`)** — the library itself; `src/main/java` plus `src/autogenerated/java` (registry data generated by `code-generators`, do not hand-edit).
- **`testing` (`net.minestom:testing`)** — integration testing framework (published artifact). See below.
- **`demo`** — runnable example server. Useful as a reference for wiring up a server, not part of the published artifact.
- **`code-generators`** — generates the `src/autogenerated` sources from minestom-data (registries, blocks, items, etc.).
- **`jmh-benchmarks`**, **`jcstress-tests`** — perf and concurrency harnesses.
- **`build-src`** — shared Gradle convention plugins (`minestom.java-library`, `minestom.publishing`, `minestom.java-binary`).

## Architecture big picture

Entry point is `MinecraftServer` which boots a `ServerProcess` (`ServerProcessImpl`). The `ServerProcess` is the composition root — every subsystem (instance/chunk manager, entity tracker, connection/network, event node, scheduler, block manager, command manager, recipe manager, etc.) hangs off it. There is no global singleton state beyond what `MinecraftServer` exposes statically; tests construct fresh `ServerProcess` instances per test.

Key concepts you will see across packages:

- **Instance / Chunk / Block** (`instance/`) — Instances replace vanilla "worlds". They're cheap, in-memory by default, and managed by a shared thread pool rather than one-thread-per-world. Blocks are data; behavior (e.g. opening a chest) requires a registered `BlockHandler`.
- **Entity** (`entity/`) — no passive/aggressive split; AI is opt-in. `Player` extends `Entity`.
- **Event** (`event/`) — hierarchical `EventNode` system, not annotation-based listeners.
- **Network** (`network/`) — packet types in `network.packet.{client,server}.*`; protocol versioning lives in `network.NetworkBuffer` and the codec layer.
- **Codec / Registry** (`codec/`, `registry/`) — registry-driven serialization. New registry-backed data usually flows through `code-generators`.
- **Thread / Acquirable** (`thread/`) — entities/chunks are tied to a thread via the acquirable system; `-Dminestom.acquirable-strict=true` in tests will throw on misuse.

`ServerFlag` and `FeatureFlag` hold tunables — prefer adding new ones there rather than scattering system properties.

## Tests: use the integration framework for anything server-touching

**Any new feature that touches a running Minecraft server (instances, entities, players, packets, events, world interactions, commands, inventory, etc.) MUST be covered by an integration test using the `testing` module.** A plain JUnit unit test is only appropriate for pure utilities with no `ServerProcess` dependency (parsers, math/coordinate helpers, codecs in isolation, data structures).

Integration test shape:

```java
@EnvTest                                    // testing/src/main/java/net/minestom/testing/EnvTest.java
public class MyFeatureIntegrationTest {
    @Test
    public void something(Env env) {        // Env is injected by EnvTestExt
        var instance = env.createFlatInstance();
        var player = env.createPlayer(instance, new Pos(0, 42, 0));
        // env.tick(), env.tickWhile(...), env.trackEvent(...), env.listen(...),
        // env.createConnection() for protocol-level assertions
    }
}
```

`Env` (`testing/src/main/java/net/minestom/testing/Env.java`) gives you a fresh `ServerProcess`, helpers to spin up instances/players/connections, manual ticking, and event/packet collectors. Look at existing `*IntegrationTest.java` files for patterns (e.g. `src/test/java/net/minestom/server/snapshot/EntitySnapshotIntegrationTest.java`, `src/test/java/net/minestom/server/listener/PacketListenerIntegrationTest.java`).

Name new tests `*IntegrationTest` when they boot a `ServerProcess`; reserve plain `*Test` for unit-level coverage.

## Other conventions

- Only Jetbrains `@NotNull`/`@Nullable` annotations — `checker-qual` is explicitly excluded from the classpath.
- Generated sources under `src/autogenerated/` are committed but regenerated by `:code-generators`; if a registry/data change is needed, update `code-generators` rather than the generated file.
